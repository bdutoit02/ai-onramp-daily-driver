Getting set up

Claude lives in the cloud. Your conversations, projects, and settings are all stored on Anthropic’s servers, not on your machine — so whichever way you access Claude, you’re working in the same place, with one account. That means you can start on the web today, switch to the desktop app next week, and everything is exactly where you left it.

The default: start with the web. Go to claude.ai and create an account. Clean, safe, nothing on your local machine. This is where you’ll spend the next week.

When you’re ready: the desktop app. Download it from claude.ai. It’s the same Claude, a bit more convenient — sits in your taskbar, easier to make a daily habit. In chat mode it behaves identically to the web. You’ll notice Code and Cowork in the sidebar — leave those for now, we’ll get to them later in the course.

One practical note: if your organisation doesn’t yet have an enterprise AI agreement, use a personal account for now. We’ll cover the security and policy considerations shortly.

Subscription

Free is fine for trying Claude out. But if you’ve decided you want to build a real working relationship with it, get Pro. It’s $20 a month and it’s probably the highest-return $20 you’ll spend on your professional development this year.

Here’s what Pro gets you:

The habit argument: usage limits on Free don’t just inconvenience you, they break the habit loop at exactly the moment you’re trying to form it. Pro removes that friction entirely.

QS2: Set up your Projects

QuickStart
  1. Click on Projects in the left sidebar, then “New project”.
  2. Give it a name and a description (“what are you trying to achieve”). This description is for you and coworkers; it is not referenced by Claude.
  3. Populate Instructions. This is perhaps the key piece of information. This will guide Claude in any of the Chats created under this Project. It can be very detailed, if appropriate.
  4. Populate Files, by drag and drop or upload. These files will be referenced by every chat in the project.
Understand and Explore
Projects

By default every conversation in Claude is isolated — it doesn’t know what you discussed yesterday or last week. Projects fix that.

A Project is a persistent workspace with four elements:

  • A name and purpose — what the project is for
  • Instructions — how you want Claude to behave within this project: your role, your preferences, your rules
  • A knowledge base — files, documents, and reference material you upload once and Claude draws on whenever you need it. You can also connect GitHub repositories and Google Drive directly
  • Chats — all the conversations you have within the project, kept together in one place

What this means in practice: instead of starting every conversation from scratch, you build a Project for each significant area of your work — a client, a research topic, a recurring task. Over time the Project accumulates context and becomes genuinely useful as a working environment rather than just a chat interface.

Organising your chat history matters. A Project is only as useful as the discipline you bring to it. Name your files descriptively, keep your knowledge base current, and think of the Project as a living workspace rather than a filing cabinet. Unstructured chat history quickly becomes noise.

A note on collaboration: sharing Projects with colleagues requires a Team or Enterprise subscription. On a personal Pro account your Projects are private.

Tips
  • Instructions and files that are only relevant to a particular chat can be inserted directly into those Chats.
  • A Project also has a memory, populated by Claude as the Project evolves. But you can edit the memory yourself by adding stuff or removing stuff. While “Instructions” and “Files” are the most obvious resources available to Claude in a Project, “Memory” is a powerful tool, especially in longer projects. If you are planning to build a Project that you use repeatedly over time, spend time finding out exactly how “Project memory” works (ask Claude!).
  • You need to think carefully about exactly how a Project is going to be used. You may want to start small and let it grow. Don’t be afraid to start a new Project if you have misconceived the first iteration.
  • You can manage your Project space as it grows, by favouriting, deleting and archiving.

Understand and Explore
Artifacts

Sometimes Claude’s response isn’t just text in the conversation — it produces a self-contained piece of work: a document, a table, a piece of code, a formatted report, an interactive page. These are called Artifacts and they appear in a separate panel alongside the conversation.

Why it matters: an Artifact isn’t just a response, it’s a working document. You can ask Claude to revise it, extend it, reformat it — all while keeping the conversation going. The Artifact updates in place rather than producing a new block of text each time. You can also switch between the finished output and the underlying code if you want to see how it was built.

What triggers an Artifact: Claude decides based on the nature of the output. Longer structured documents, code, and formatted content typically appear as Artifacts automatically.

Artifacts can be shared or published directly from Claude.

Adding stuff

One of Claude’s most underused capabilities: you don’t have to explain everything in words. You can bring material directly into the conversation.

  • Typed context — paste in a document, a data extract, an email thread, a set of notes. Claude reads it and works with it. This is often faster and more accurate than trying to describe something in your own words.
  • Uploaded files — attach files directly to a conversation. Supported formats include PDF, DOCX, CSV, TXT, and common image formats like PNG and JPEG.
  • Screenshots — you can paste or upload a screenshot directly into the conversation. Useful for sharing something visual without having to describe it — a chart, a UI, a document you can’t easily export.

The practical habit: before you ask Claude to help with something, ask yourself whether there’s a document, file, or screenshot that would give it better raw material than your description alone. The easiest and most intuitive methods are simply to copy and paste from your clipboard, or drag and drop a file, directly into the conversation.

Tips
  • When Claude produces an Artifact, don’t just copy and paste it and move on. Stay in the conversation and refine it until it’s actually what you need.
  • Even if Claude has decided not to show something as an Artifact, you can ask it to.
  • Artifacts are “updated in place” — so there is only one version on display. But there is an updated version number and you can go back to earlier versions.
  • Artifacts can also be interactive. For this and other reasons, they can be used as mini-demos, in place, without having to create a demo page.
  • As the course progresses, you’ll pick up more powerful ways to bring material into Claude: Connectors give Claude persistent access to cloud services; Cowork and Code give Claude direct access to approved local directories. These become important when you’re working with large datasets or information that changes frequently. For now, copy/paste and drag/drop will take you a long way.
  • Note that as you start to use Projects, persistent memory, and Skills, Claude will often know a lot of your context before you even start a new chat.

Connectors

So far everything you’ve brought into a conversation you’ve had to actively add — typed context, uploaded files. Connectors are different: they give Claude persistent access to external services so the information is already there when you need it.

Connectors don’t just let Claude read from external services — they allow Claude to act on your behalf too. That’s the shift from assistant to informed collaborator.

There are two types:

  • Web connectors — connect cloud services like Google Drive, Gmail, and Google Calendar. Set up once in Settings, available across all your chats and Projects
  • Desktop extensions — connect Claude to things on your local machine: your file system, local applications, and tools like the Chrome extension that lets Claude see your open tabs. These are available through the desktop app

  • There are a lot of Connectors available and learning how to use them is complex. Start with one and don’t rush into it. Gmail is a good choice. Hook it up and ask Claude some questions about your email history (“Review all the important emails in the last month that seem to require some kind of response”).
  • To check that a connector is properly set up and active, you can just ask Claude.

Understand and Explore
Basic chat technique

The single biggest mistake new users make: type a question, read the answer, close the chat. That’s using Claude as a search engine. It’s not what it’s for.

Three things to give Claude before every task:

  • Context — who you are, what you’re working on, why it matters
  • Task — what you actually want Claude to do
  • Guidelines — tone, format, length, constraints

You don’t need all three every time, but the habit of thinking in these terms immediately improves your results.

Stay in the conversation. Claude’s first response is a starting point, not a finished product. Push back, redirect, ask for alternatives. Sometimes, start from scratch in a new chat. The value compounds as the conversation develops.

Fluency is not accuracy. Claude writes clearly and confidently even when it’s wrong. A polished answer is not a correct answer. Verify anything that matters.

One last thing

Open Claude before you open anything else in the morning.

Not because you’ll always have something specific to do with it. But because the habit of reaching for it first — before email, before your task list, before your browser — is what turns an occasional tool into a working partner.

The people who get the most out of Claude aren’t the ones who use it for the biggest tasks. They’re the ones who use it constantly, for small things and large, until the instinct to reach for it becomes automatic.

AI Security Essentials

A take-home guide for course participants

Using AI tools like Claude as a daily driver is low-risk for most people — but a few simple habits make it genuinely safe. This guide separates what actually matters from what you can stop worrying about.

1. External Threats — Lower Risk Than You Think

When you use a web-based AI tool like Claude.ai, you are trusting a well-resourced technology company to handle security on your behalf. Counterintuitively, this is safer than managing software locally — you are not responsible for keeping anything patched, configured, or updated.

Your conversation data is transmitted over an encrypted connection (TLS), so interception in transit is not a realistic risk. Anthropic’s servers are not a likely target for someone trying to steal your specific data.

The real external risks are account-level, not platform-level:

  • Phishing — fake login pages designed to steal your credentials
  • Password reuse — if one account is breached, others become vulnerable
  • Device compromise — malware on your machine captures data before it reaches any server

Action: Two things to do now

  • Install Bitwarden (free, open source). One strong master password protects all your accounts, and you stop reusing passwords across AI tools and other services.
  • Enable two-factor authentication (2FA) on your Claude.ai account. This prevents account takeover even if your password is stolen.

2. Confidential Data — This Is Where Judgement Matters

When you paste text into an AI chat window, you are sending it to a third-party server. For most personal use this is fine. For work data, it requires thought.

Ask yourself one question before pasting anything:

“Would I be comfortable if my employer saw exactly what I just pasted into this chat?”

If the answer is no — or even maybe — use made-up data instead. You can learn everything you need to learn with synthetic examples. Real work data adds risk and is rarely necessary.

Things to be aware of:

  • Many employers now have AI use policies. Check yours. Breaching it could have professional consequences regardless of any actual data leak.
  • Regulated industries (healthcare, legal, finance, government) have specific data handling obligations that may apply even to informal use.
  • By default, AI providers may use conversations to improve their models. You can opt out of this in Claude.ai settings — worth doing as a matter of routine.
  • If your organisation needs to use AI with sensitive data legitimately, enterprise tiers (such as Claude for Work) have different contractual data protections.

3. A Built-In Safety Prompt

Claude.ai lets you set a permanent system prompt — background instructions that apply to every conversation. As part of your setup, we will add an instruction that prompts Claude to flag inputs that look like real personal or work data (lists of names with numbers, email addresses, client records, etc.) and ask you whether you want to continue.

This is not a technical guardrail — it will not catch everything. But it creates a useful pause at exactly the moment you might otherwise act without thinking. Think of it as a seatbelt reminder chime, not a seatbelt.

Your Security Checklist

  • Install Bitwarden and create a strong master password
  • Enable 2FA on Claude.ai (and any other AI accounts you create)
  • Check the training data opt-out setting in Claude.ai preferences
  • Find out if your employer has an AI use policy
  • Default to synthetic or anonymised data when practising — not real work data
  • Apply the one-question test before pasting anything sensitive

Yes, this is getting somewhere good. Let me break down what you're describing as a source discovery pipeline, because I think there are actually four distinct moves hiding in your description:

Move 1: DR sweep. Run Deep Research on the question. Harvest the source list. Discard the report. This is clever -- you're using DR as a search engine with judgment, not as a report writer. The sources it finds will reflect its real-time web access and its own assessment of relevance.

Move 2: Training knowledge sweep. Give a standard Claude chat the question and the DR source list, and ask: what's missing? This taps the model's knowledge of authoritative institutions, recurring reports, key authors, regulatory bodies -- things it knows exist from training even if DR didn't surface them in its web search. This is a genuinely different knowledge base from Move 1 and the overlap won't be complete.

Move 3: Human input. The researcher adds sources they know about -- industry contacts' reports, internal documents they've seen referenced, niche publications the AI wouldn't prioritise. This is also where the Session 3 homework could land naturally: participants arrived having identified weaknesses in the canned reports, and some of those weaknesses were source gaps. Their own knowledge of the domain feeds in here.

Move 4: Quality check. An AI pass over the combined list to verify: is the URL live, how recent is the publication, is this a primary source or a secondary summary, is the authoring institution credible? This is triage before the human does final curation.

Then the human edit -- the A-D ranking, removals, additions.

A few reactions:

1. The Skilled User Frame

Session 3: Deep Research in the Wild

As AI tools become more capable, the gap between casual users and skilled users widens. Skilled users are not necessarily more technical — they are more deliberate. They think clearly about three things:

Input — the material you bring to the conversation: documents, data, files, screenshots. The raw substance the model works with.

Direction — everything you say to the model: the question, the context, the instructions, the constraints, the format, the iteration. Direction is how you guide the model toward useful work.

Evaluation — how you assess what comes back. Not “does this look good?” but structured, criteria-driven judgment applied consistently.

This session focuses on Evaluation. Direction becomes the focus of Session 4. Input becomes the focus of Session 5.

A note on why evaluation matters: AI outputs are increasingly polished and confident in tone. The easier they are to read, the harder it is to spot what is missing, overstated, or wrong. Evaluation is the skill that keeps you in control of the work — and it transfers across every tool you will ever use.

2. Key Concepts

Session 3: Deep Research in the Wild

Deep Research

Deep Research is a specific product feature — available in Claude, ChatGPT, and Perplexity — not simply a description of researching with AI. It was among the first applications outside of coding to combine three capabilities that had previously operated separately:

  • Extended thinking
  • Agentic behaviour
  • Tool use

When you trigger Deep Research, you are not sending a prompt to a language model and waiting for a response. You are initiating an autonomous research process that can run for several minutes, visit dozens of sources, and make hundreds of decisions before producing its output.

Extended Thinking / Chain-of-Thought Reasoning

Standard AI responses are generated fluidly, with no visible reasoning step. Extended thinking changes this: the model is given time and compute to plan its approach, break the problem down, and reason through it before producing output. Where visible — as in Claude’s thinking indicator or ChatGPT’s activity log — you are watching the model work, not just watching it write.

This is inference-time computation: the model doing more work per question rather than simply being a larger model.

Agentic Behaviour

An agent is a system that pursues a goal by taking a sequence of actions, using tools, and making decisions along the way. Deep Research tools behave agentically: they decide what to search for, choose which sources to read, extract what seems relevant, identify gaps, and search again. They are not following a fixed script — they are planning and replanning in response to what they find.

Tool Use

Agentic behaviour requires tools. In Deep Research, the primary tool is web search: the model calls the search tool, receives results, decides what to do with them, and calls it again as needed. The search–analyse–replan cycle — visible in ChatGPT’s activity log — can repeat dozens of times in a single research run. This is the same architecture that underlies the more powerful agentic systems we will see in Session 6.

3. The Evaluation Rubric

Process Quality — how did it research?

  • Transparency of reasoning — can you see what it did, what it chose, and why?

Content Quality — what did it find?

  • Quality of insight extraction — did it draw meaningful conclusions from its sources, or summarise superficially?
  • Domain coverage — did it address the full scope of the question, or leave significant areas thin or absent?
  • Contextual accuracy — did it handle the specific regulatory, institutional, and market context correctly?
  • Logical coherence — do the findings hold together as an argument, or sit as disconnected observations?

Output Quality — what did it produce?

  • Structure and navigability — is the report organised clearly and easy to use?
  • Specificity of recommendations — are recommendations actionable, or generic enough to apply to any situation?
  • Caveats and limitations — does the report acknowledge what it does not know, where data is incomplete, or where claims are uncertain?
  • Fitness for purpose — could you hand this report to a senior client and defend every claim in it?

That last question is the one that matters most for the kind of work this course is built around. A report that looks authoritative but cannot survive scrutiny is not a research output — it is a liability.

1. Where We Are

In Session 3 you watched three canned deep research tools tackle the same question and saw that their outputs -- however polished -- contained hidden weaknesses that only systematic evaluation exposed. The homework question was: what would you need to change, add, or verify before you could stand behind one of those reports?

This session answers that question. It walks through a complete directed research process -- from source discovery to a finished report -- designed so that every step is transparent, every source is chosen deliberately, and the final output is defensible under scrutiny.

In IDE terms, Session 3 focused on Evaluation. This session focuses on Direction. You will see that Direction is not a single skill but a repertoire -- different stages of a research process need different kinds of instruction, and understanding why each instruction is shaped the way it is matters more than memorising templates.

2. The Process Architecture

The research process demonstrated in this session has eight steps, organised as a funnel: wide at the top (casting a broad net for sources), narrowing through quality checks and human curation, synthesised into themes, and shaped into a defensible report.

  • DR sweep -- Run Deep Research on the question. Harvest the source list. Discard the report.
  • Training knowledge sweep -- Ask Claude to identify authoritative sources missing from the DR list, drawing on its training knowledge.
  • Human knowledge sweep -- The researcher adds sources from their own domain expertise.
  • Source quality check -- Claude checks each source for accessibility, recency, credibility, and relevance. Output: an annotated audit.
  • Human curation -- The researcher reviews the audit, ranks sources, and produces a clean curated list.
  • Literature review -- Claude reads and summarises each curated source individually, focused on the research question. No synthesis yet.
  • Analytical synthesis -- Claude organises the evidence by theme, rates evidence strength, surfaces tensions and gaps. No recommendations yet.
  • Report -- Claude produces a structured, evidence-graded report shaped by a detailed instruction specifying audience, structure, and evidence-handling rules.

Each step produces a distinct product that feeds the next. Each requires a different kind of instruction. The process can be run end to end in a Claude Project, with all intermediate products preserved as a transparent audit trail.

3. Source Discovery and Curation

Canned deep research tools select their own sources autonomously. You have no control over what they find, what they miss, or what they prioritise. The source discovery pipeline gives you that control.

3.1 -- Deep Research sweep

Run Deep Research on the research question with minimal direction. The purpose is to use DR as a search engine with judgment -- it will visit dozens or hundreds of pages and identify the ones it considers most relevant. When it finishes, extract the list of sources it cited. The report itself is not the product of this step; the source list is.

In the session demo, DR cited 73 sources (while visiting over 500 pages during its search). The distinction between pages visited and sources cited is worth noting -- the large number displayed during the run does not mean all of those pages ended up in the report.

3.2 -- Training knowledge sweep

In a standard Claude chat (not DR), provide the research question and the DR source list. Ask Claude to identify authoritative sources that are relevant but missing from the list. This taps a different knowledge base: the model's awareness of institutions, recurring reports, key authors, and regulatory bodies from its training data, which may not have appeared in DR's real-time web search.

In the session demo, this sweep produced approximately 40 additional sources and identified three significant gaps in the DR list: the Big 5 banks' own annual integrated reports were entirely absent, some sources cited in the DR report's body text did not appear in its own source list, and an entire category of practical data (recruitment firm salary guides) was missing.

3.3 -- Human knowledge sweep

The researcher adds sources from their own domain knowledge: reports they have seen cited, institutions they know are active in the space, niche publications, known experts. This is where the researcher's professional expertise enters the process. No AI can substitute for deep domain familiarity.

3.4 -- Source quality check

Give Claude the combined source list (from all three sweeps) and ask it to check each source for: URL accessibility, publication date, whether it is a primary or secondary source, institutional credibility of the publisher, and direct relevance to the research question. Ask for the output as a structured table (CSV or spreadsheet) with a recommendation column: Keep, Query, or Remove.

In the session demo, 115 sources were checked. Key findings: 61 recommended to Keep, 40 to Query, 14 to Remove. Only 62 URLs were confirmed live. Of the 42 training knowledge sources, only 18 had confirmed live URLs -- confirming that training knowledge produces good institutional names but unreliable URLs.

A practical note: checking 115 URLs one by one hits Claude's tool call limits. The workaround is clicking "continue" when prompted. For production use, a script that handles mechanical URL checks and reserves AI for judgment tasks (credibility, relevance) would be more efficient.

3.5 -- Human curation

The researcher reviews the annotated source audit and makes the final decisions: which sources to keep, which to discard, and how to rank them. A simple ranking system works well: A (essential -- must be covered in depth), B (important -- should be covered), C (useful -- reference if relevant), D (ignore).

In the session demo, a tight filter was applied: keep only sources with Status = Live and Recommendation = Keep. This produced 40 sources with an even split of 20 primary and 20 secondary sources -- a clean, defensible list.

This is the step where human judgment is irreplaceable. The AI can check URLs and assess credibility, but deciding which sources matter most for this specific question and this specific audience is a human call.

The product: A transparent, quality-checked, human-curated source list. Every source is there for a reason. Every source has been verified. If a client asks "why did you use this source?" or "why didn't you use that one?", you can answer.

A note on recurring research: The source list becomes a maintained asset. The expensive multi-sweep discovery runs once. On subsequent runs -- monthly, quarterly -- the task is simpler: check the canonical list for health and search for new publications from known institutions. The process gets faster and the list gets better over time.

4. Literature Review

With the curated source list in hand, the next step is to find out what the sources actually say. The instruction asks Claude to read each source and produce a focused summary, concentrating on content relevant to the research question. Crucially, it asks for individual summaries -- no integration, no synthesis, no attempt to answer the research question. The product is a structured digest: what each source says, on its own terms.

This step requires web access (Claude needs to visit the URLs and read the content). In the session demo, DR mode was used for this purpose. The instruction explicitly told DR not to search for additional sources beyond the curated list. DR acknowledged the constraint but visited over 400 pages during the run -- it did not fully respect the process constraint, although the final output reported on only the 40 specified sources.

The lesson: DR may not respect process constraints even when it respects output constraints. For production work where strict source control matters, standard chat with web search may give more reliable control.

5. Analytical Synthesis

The synthesis is the analytical backbone of the process. It takes the 40 individual source summaries and reorganises them by theme, answering four questions:

  • What are the major themes that emerge across the sources? Which sources support each theme, and how strongly?
  • Where do sources present conflicting data, different figures, or divergent assessments of the same issue?
  • What significant gaps remain in the evidence base -- areas the research question requires but the sources do not adequately address?
  • Where is the evidence strong (multiple authoritative primary sources converging) versus where does it rest on a single source or on secondary reporting of the same underlying data?

The instruction for the synthesis explicitly states: do not make recommendations. Organise and assess. This separation matters. If you go straight from source summaries to a report, the AI will do the synthesis implicitly and you lose visibility into the analytical layer. You cannot see whether a recommendation is well-supported or built on a single data point.

Key findings from the session demo synthesis:

  • 11 themes identified, each with an explicit evidence-strength rating
  • Cybersecurity was the best-evidenced shortage, supported by multiple independent primary sources. But the specific figure of a 30,000-person deficit came from a single paywalled commercial report whose methodology could not be verified.
  • AI/ML demand had strong directional consensus from bank spending data and hiring patterns, but no source provided a quantified estimate of the gap
  • Two articles that appeared to be independent sources were authored by the same organisation (Fortinet) citing the same underlying data -- what looked like corroboration was actually a single evidence stream
  • IT spending figures for one bank appeared at four different values across sources, due to different reporting boundaries (with/without staff costs, group vs. SA operations) -- flagged as a tension, not silently resolved
  • Nine significant evidence gaps were identified, including the absence of the Big 5 banks' own annual reports, salary benchmarking data, and education pipeline data

The synthesis distinguishes between what the evidence supports, what it suggests but cannot confirm, and what it does not address at all. That distinction is what makes the final report defensible.

6. The Report

The report instruction is the most detailed and opinionated in the process. It specifies:

  • Audience and purpose -- who will read this and what it is for. In the demo: a senior leadership team at a Big 5 bank, with the report demonstrating analytical capability.
  • Explicit structure -- the sections the report must contain, each with a clear function. In the demo: executive summary, skills landscape, regulatory drivers, supply-side problem, bank responses, evidence gaps, and a prioritised training framework.
  • Evidence-handling rules -- where evidence is strong, say so. Where it rests on a single source, say that too. Do not present weak evidence with the same confidence as strong evidence. Where the synthesis identified tensions, acknowledge them rather than resolving them silently.
  • The no-filler rule -- no generic paragraphs about the importance of digital transformation. Every paragraph must contain a specific claim, a specific piece of evidence, or a specific judgment.

The report produced in the session demo ran to approximately 11 pages. It ranked five skills shortage areas by evidence strength and strategic urgency. It presented a regulatory drivers table covering six simultaneous regulatory changes with timelines and evidence ratings. It acknowledged what the analysis could not tell us. And it closed with a substantive argument about the kind of training investment the evidence supports.

The critical comparison with Session 3: The canned reports from Session 3 are slicker and read more fluently. But this report knows what it does not know. It distinguishes strong evidence from weak. It surfaces contradictions rather than hiding them. It has a transparent, auditable source base. It is defensible under scrutiny -- and the canned reports are not.

That standard -- defensible under scrutiny -- is the core argument of this session. Everything in the process contributes to it: the multi-sweep source discovery catches what any single tool misses, the quality audit removes unreliable sources before they can contaminate the analysis, the human curation applies domain judgment, the separated synthesis makes the evidence structure visible, and the evidence-graded report inherits all of that transparency.

7. The Direction Repertoire

One of the key lessons of this session is that Direction is not a single skill. Different stages of a research process require fundamentally different kinds of instruction. The process demonstrated today used six distinct types:

  • Minimal direction (DR sweep) -- deliberately under-specified, to see what the tool finds on its own without constraining the search
  • Knowledge elicitation (training knowledge sweep) -- asking Claude to critique a list against its own training, drawing out knowledge that would not surface through web search
  • Structured audit (source quality check) -- explicit column definitions, a specified output format, and actionable recommendations for each item
  • Constraint instruction (literature review) -- telling Claude what not to do: do not add sources, do not synthesise, do not answer the question
  • Analytical instruction (synthesis) -- organise and assess the evidence, surface tensions and gaps, but do not draw conclusions or make recommendations
  • Opinionated instruction (report) -- a detailed specification of audience, structure, tone, evidence-handling rules, and quality standards

Each instruction is shaped by what that step needs to produce. A minimal instruction would be wrong for the report. An opinionated instruction would constrain the DR sweep prematurely. Understanding why each instruction is shaped the way it is -- not just what it says -- is the skill that transfers across every research process you will design.

A Practitioner’s Guide to LLMs

What They Are and How to Enhance Them

Opening

To use any tool well, you need an accurate mental model of what it is. This is especially true for LLMs, because the two most common mental models — “it’s just autocomplete” and “it’s a superintelligence” — both lead to predictable mistakes. We’re going to build a better one.

The structure is simple: first, what kind of intelligence is an LLM? Then, how do we enhance it?

A. What Kind of Intelligence Is an LLM?

A1. It’s a neural network

A neural network is a system of interconnected computational units, loosely inspired by neurons in the brain. Early AI systems — chess engines, expert systems — were rule-based: a developer wrote explicit rules, and the machine followed them. Neural networks work differently: instead of following rules, they learn from examples. Through a process called gradient descent, the system repeatedly adjusts its internal connections, taking small steps in the direction of better performance — like finding the lowest point in a hilly landscape by always stepping downhill — until it gets reliably better at the task.

This is deep learning — neural networks with many layers, where each layer learns to recognise increasingly abstract patterns. Early layers might detect edges in an image; deeper layers recognise faces. The key insight that unlocked practical deep learning was not new mathematics — it was scale. When you combine this architecture with enough data and enough computing power, something remarkable happens: capability emerges that nobody explicitly programmed.

A2. It’s a language model

The second thing to understand is what this particular neural network was trained to do — and what that training produced.

The story begins with machine translation. In the 2010s, neural networks trained on parallel texts in different languages began outperforming rule-based translation systems. Researchers noticed something interesting: to translate well, the model had to develop rich internal representations of meaning, not just word-for-word mappings. Language was turning out to be a surprisingly powerful training signal.

The decisive architectural breakthrough came in 2017. A paper called Attention Is All You Need introduced the attention mechanism and the Transformer architecture — a design that lets the model consider every word in a passage in relation to every other word simultaneously, rather than processing language sequentially. Because attention allows for parallel processing in a way that pre-Transformer models did not, this made it possible to train on vastly larger datasets in parallel. You could go very big, very fast.

What followed was a decision to train these networks on an enormous fraction of the written internet — books, articles, code, conversation. The result is a model that has learned the statistical distribution of human language: what kinds of text follow from what other kinds of text, across essentially the full breadth of recorded human knowledge.

This is the key conceptual move: an LLM is not a database. It doesn’t store facts and retrieve them. It has compressed the patterns of an enormous corpus into a set of internal weights, and when you prompt it, it generates a response by sampling from those learned patterns — predicting what a good continuation would look like.

This is genuinely powerful. Predicting what a good answer looks like, across the full range of human knowledge, requires rich internal representations of that knowledge. But it is also the source of the model’s most important failure mode: hallucination. The model isn’t lying when it produces a plausible-sounding but false citation. It has no mechanism to distinguish “I know this” from “this fits the pattern of something I’d know.” It generates fluently in both cases. Confidence is not a signal of accuracy. This is not a bug to be patched — it is a property of the architecture. The right response is not to distrust the tool, but to understand when verification matters.

Bridge

So: a neural network, trained on human language, that models statistical distributions rather than storing facts. This base intelligence has characteristic strengths — fluency, breadth, pattern recognition across domains, tireless availability — and characteristic weaknesses — hallucination, no persistent memory, no reliable self-knowledge about the limits of its knowledge.

Enhancement is the natural response to that profile. And the idea of enhancing a base intelligence is not new. We have always built on top of the base intelligence we were given — through language, writing, mathematics, computation, and now AI. Each layer extended what came before. The same logic applies here.

A brief aside — and then back to work

We should note something that is easy to miss in the rush to use these tools: we now live alongside a genuinely new kind of intelligence. Not a smarter human. Not a dumber human. Something built on entirely different foundations — designed rather than evolved, trained on human artefacts rather than shaped by survival pressures, with a completely different profile of strengths and weaknesses.

We have, until very recently, shared the world only with other evolved intelligences — humans and animals, all shaped by the same basic pressures. Artificial intelligences are categorically different. They are behaviourally familiar, because they were trained on everything we wrote, but structurally alien.

The most useful approach is not to rank them against human intelligence, but to understand them on their own terms. What is this thing actually good at? What does it genuinely struggle with? That’s the question that makes you a better user — and it’s the question the rest of this session is designed to answer.

B. Enhancing the Intelligence

B1. Model enhancements — what the builders do

The base model — trained on the internet, predicting next tokens — is powerful but raw. It will complete any text in whatever direction the patterns suggest, including harmful, false, or simply unhelpful directions. Several layers of enhancement are applied before you interact with it.

Reinforcement learning from human feedback (RLHF) and related fine-tuning techniques shape the model’s behaviour: making it helpful, honest, and safe. Human raters evaluate outputs; the model learns to produce outputs that score well. This is why Claude behaves like an assistant rather than a raw text predictor.

Explicit reasoning — sometimes called chain-of-thought — is the discovery that models perform dramatically better on complex problems when they are prompted, or trained, to think step by step before answering. Rather than jumping to a completion, the model works through intermediate steps. This is the basis of the “extended thinking” modes you may have encountered.

Inference-time scaling is the most recent development: rather than just training bigger models, you run more computation at the moment of answering — letting the model explore multiple reasoning paths, check its own work, or run longer internal deliberation. This has driven significant capability jumps in the last year.

Most recently, agentic capacity itself is becoming a builder-level enhancement — shipped as part of the foundation model rather than constructed by users. Claude Code, which autonomously plans and executes complex coding tasks, and Cowork, which automates file and task management for non-developers, are current examples. The boundary between “what the model can do” and “what you have to build” is shifting rapidly — a point we will return to.

B2. User techniques — what you can do

Context is the most important and most underused lever. The model has no persistent memory — everything it knows about your situation is what you put in front of it in the current conversation. Rich context produces dramatically better output. Most new users underinvest here.

Prompt engineering is the practice of structuring your inputs to guide the model’s thinking and outputs. It ranges from simple (be specific, give examples, specify the format you want) to sophisticated (role assignment, chain-of-thought prompting, system prompts).

Tools and skills extend the model beyond its training data. A model with access to web search, code execution, or file systems can act in the world, not just generate text.

Agents are systems where the model plans and executes multi-step tasks autonomously — breaking a complex goal into steps, using tools along the way, and iterating until the job is done. This is where the technology is moving fastest, with agentic capabilities increasingly being built into the underlying models and agentic applications such as Claude Code.

Coda: Why Understanding Matters More Than Technique

There is a pattern worth naming before we finish.

Since ChatGPT went public in late 2022, the capability of foundation models has risen steeply and continuously. Things that required significant custom engineering 18 months ago are now handled natively — by the model itself, or by tools like Claude Code, or by agents that no longer need elaborate custom infrastructure to orchestrate. People and organisations that invested heavily in building technical workarounds for the limitations of 2023-era models have repeatedly found that investment overtaken by model improvements. The specific workaround becomes unnecessary. The Python pipeline someone spent months building gets replaced by a conversation.

This creates a real risk for anyone trying to develop AI capability: following the hype into deep investment in today’s limitations, only to find the model outgrows them in six months.

The antidote is literacy. Understanding what the model actually is — how it learns, what it’s good at, where it fails and why — compounds over time. It makes you a better user of today’s models and a faster adapter to tomorrow’s. Specific techniques become obsolete. The understanding that lets you evaluate and adopt new techniques does not.

That is what this course is designed to build. Not a toolkit for 2026. A foundation that gets more valuable as the models get more capable.
